# NextChat UI 延续指南

## 这份文档的目的

这不是 NextChat 官方提供的完整设计系统文档，而是根据当前代码实现整理出的 UI 延续指南。

它的目标只有两个：

1. 帮后续维护者快速理解 NextChat 现在的界面风格。
2. 帮后续 AI 在不跑偏的前提下继续设计和开发新界面。

## 结论先说

NextChat 目前有一套相对稳定的 UI 风格，但它主要存在于代码里，而不是存在于独立的设计文档、Figma、Storybook 或设计系统站点中。

因此，后续 AI 可以继续沿用它的 UI 设计，但前提是必须把当前代码里的视觉规则当作约束，而不是自由发挥。

## 设计系统现状

### 已存在的部分

- 有明确的浅色 / 深色主题变量
- 有统一的窗口式主布局
- 有统一的基础组件风格
- 有统一的圆角、边框、阴影和主色
- 有统一的侧边栏、列表、弹窗、按钮、Toast 等模式
- 有移动端适配逻辑

### 缺失的部分

- 没有独立的视觉规范文档
- 没有 Figma 设计源文件入口
- 没有 Storybook 或组件展示站
- 没有正式命名的 spacing / typography token 文档
- 没有一套对外明确发布的设计原则说明

## 当前 UI 的核心视觉语言

### 1. 整体结构

NextChat 的主界面不是传统网页落地页，而是“桌面应用窗口”风格：

- 页面主体是一个居中的大容器
- 左侧是功能 / 会话侧边栏
- 右侧是主要内容区域
- 在桌面端强调窗口边界、圆角和阴影
- 在移动端切换为更贴近全屏 App 的布局

这种结构是整个产品最重要的视觉识别点之一。

### 2. 颜色系统

当前项目使用 CSS 变量做主题切换，主变量在 `app/styles/globals.scss` 中。

主要特征：

- 主色是偏蓝绿的 `--primary`
- 浅色主题以浅灰、白底、低对比边框为主
- 深色主题不是纯黑，而是带一点灰蓝感的暗背景
- 侧边栏背景和主内容区背景有轻微层次差异

后续新增界面时：

- 优先复用已有变量
- 不要新增一套完全不同的品牌色
- 不要突然把界面改成高饱和、霓虹、玻璃拟态或卡片瀑布流风格

### 3. 圆角、边框、阴影

项目整体是偏柔和的圆角卡片风格：

- 常见圆角：`10px`、`12px`、`20px`
- 常见边框：细边框、低对比度
- 常见阴影：轻量卡片阴影，不追求强烈悬浮感

后续新增组件时，应继续保持：

- 小控件小圆角
- 容器和主窗口中等圆角
- 阴影轻，不要做厚重投影

### 4. 字体与文字感觉

默认字体栈偏系统 UI 与中日韩兼容：

- `Noto Sans`
- `SF Pro`
- `PingFang SC`
- `Helvetica Neue`
- `Arial`

视觉感受上更接近“系统工具 / 实用型产品”，而不是品牌营销站点。

所以后续 AI 不应：

- 擅自引入强风格展示型字体
- 把页面改成杂志感、海报感或品牌大片感
- 把正文排版做得过于花哨

### 5. 动效风格

当前动效非常克制，主要是：

- 淡入
- 轻微上下位移
- hover 时的小范围状态变化

整体原则是“有反馈，但不喧宾夺主”。

后续新增动效时：

- 继续以 `opacity`、`transform` 为主
- 保持轻量、快速、实用
- 不要引入过多连续动画、复杂弹性动画或炫技转场

## 当前 UI 的布局规则

### 1. 主窗口

主窗口是产品最核心的布局单位，具有以下特点：

- 居中显示
- 固定视口比例感
- 桌面端有圆角、边框、阴影
- 移动端去掉厚重容器感，改为全屏

新页面如果属于主应用的一部分，应尽量嵌入当前窗口框架，而不是跳出成完全不同的页面骨架。

### 2. 侧边栏

侧边栏是稳定的导航区，包含：

- 品牌标题与 logo
- 顶部快捷按钮
- 会话列表或功能列表
- 底部操作区

它支持：

- 宽度拖拽
- 收窄模式
- 移动端滑出

因此后续 AI 设计侧边栏扩展能力时，应默认保持这个模型，不要随意改成顶部导航栏或多列复杂导航。

### 3. 列表

NextChat 大量使用列表式信息组织：

- 设置项列表
- 会话列表
- 选择器列表
- 配置面板列表

常见模式是：

- 标题 + 子标题
- 左侧图标 + 文本
- 右侧操作控件
- 卡片式或分组式排列

新功能如果也是“可配置项集合”，优先用列表式，不要硬改成营销卡片宫格。

## 当前 UI 的基础组件模式

### 1. 按钮

按钮以 `IconButton` 为核心，特征是：

- 小型、紧凑
- 图标优先
- 需要时再带文字
- 主要状态有默认、主按钮、危险按钮
- 常搭配边框或轻阴影

后续新增按钮时：

- 优先扩展现有按钮组件
- 保持统一高度、圆角和内边距逻辑
- 不要为每个页面重新发明按钮样式

### 2. Card / List / ListItem

这些组件已经承担了大量“通用容器”的职责，适合用在：

- 设置页面
- 选择弹层
- 结果清单
- 信息区块

如果新页面需要结构化展示，优先复用这些模式。

### 3. Modal

弹窗样式已较稳定，特点是：

- 白色或深色主题容器
- 顶部标题栏
- 底部操作区
- 可关闭
- 部分场景支持最大化

新弹窗应继续沿用这一框架，而不是引入完全不同的抽屉或第三方弹层视觉。

### 4. Toast

Toast 采用轻量圆角胶囊风格，适合短反馈。

后续 AI 设计通知反馈时：

- 短反馈优先沿用 Toast
- 不要频繁引入体积很大的全宽提示条

## 推荐的设计延续原则

后续 AI 如果要继续设计 NextChat，建议遵守以下原则：

### 必须保持

- 保持窗口式产品骨架
- 保持 sidebar + content 的主布局逻辑
- 保持现有主题变量体系
- 保持现有主色与浅/深色关系
- 保持圆角、边框、阴影的整体量级
- 保持基础组件优先复用
- 保持“工具型产品”的克制风格

### 可以优化

- 可访问性
- 焦点态
- 表单标签清晰度
- 移动端交互细节
- 长内容的截断与换行
- 状态反馈的一致性
- 组件命名与样式收敛

### 不建议擅自改变

- 不要切换到 Tailwind/shadcn 风格并混用两套视觉语言
- 不要改成营销官网风格
- 不要改成强玻璃拟态或强渐变风格
- 不要引入很多与现有风格无关的卡片组件
- 不要大面积重做导航结构
- 不要突然使用新的高识别品牌字体

## 当前项目的已知问题

这部分不是让后续 AI 照抄，而是提醒它“沿用风格”和“修正问题”要分开。

当前代码里有一些实现层面的不足，例如：

- 部分焦点态较弱
- 存在直接去掉 outline 的情况
- 存在 `transition: all`
- viewport 中限制了缩放

因此后续 AI 的正确做法应是：

- 继续沿用现有视觉语言
- 但逐步修复可访问性和交互细节问题
- 修问题时尽量不破坏当前整体外观

## 给后续 AI 的执行规则

如果你后续把任务交给另一个 AI，可以直接要求它遵守下面这些规则：

1. 先阅读 `app/styles/globals.scss`、`app/components/home.module.scss`、`app/components/ui-lib.tsx`、`app/components/button.tsx`。
2. 新 UI 必须优先复用现有组件和样式变量，不要新造一套视觉系统。
3. 新页面必须延续 NextChat 当前的窗口式布局和侧边栏逻辑。
4. 允许优化可访问性、表单、焦点态和响应式细节，但不要改变主视觉方向。
5. 除非任务明确要求，否则不要引入 Tailwind、shadcn、Radix、MUI 或新的视觉框架。
6. 做任何 UI 改动前，先说明它与现有风格的一致点，再开始实现。

下面这段可以直接作为后续 AI 的工作提示：
